# 架构审查指令：职责单一原则（SRP）排查与标记

## 0. 任务性质

这不是一次性重构任务，而是一次**全量排查 + 建档**任务。你的目标不是当场把所有问题改完，而是：

1. 系统性扫描代码库中所有违反"职责单一原则"的类、函数、模块；
2. 将每一处违规**记录进一份持久化的审查文档**，形成可追溯、可分派、可复查的清单；
3. 该文档将作为后续所有重构工作的唯一依据——没有被记录在案的问题，视为未发现；已记录但未标记为"已解决"的问题，视为架构负债持续存在。

**不允许**边扫描边顺手改代码。本任务的交付物是文档，不是代码变更。发现问题就记录，不擅自修复。

---

## 1. 核心原则（不可协商）

- **职责单一（SRP）和模块化不是"最佳实践建议"，是硬性验收标准。** 任何一个类、函数、文件，只要承担了两种以上不相关的职责，一律视为**失败的架构**，不接受"能跑就行"的借口。
- 判断"是否单一职责"的方法：**用一句话描述这个类/函数在做什么**。如果这句话里出现"和"、"以及"、"同时"、"并且"，或者需要用"、"分隔多个动词，它就不是单一职责，必须记录为违规项。
- 模块化的检验标准：**这个模块能否在不理解其他模块内部实现的前提下，被单独阅读、单独测试、单独替换？** 做不到，就是耦合过重，必须记录。
- 排查目标是**找全、记清**，不是"看起来问题不大就跳过"。凡是命中第 2 节任一条标准的，无论修改成本高低，一律入档。

---

## 2. 违规判定标准（逐条对照，命中即记录）

### 2.1 God Object / God Function
- 一个类同时管理：UI 布局构建 / 数据加载与文件系统操作 / 业务逻辑分发 / 事件处理 / 线程调度 / 统计计算，六项中占三项以上 → 记录为 God Object。
- 单个函数超过 100 行，或分支数超过 8 个，且每个分支执行的是完全不同类别的业务 → 记录为 God Function。
- 一个事件处理器混合了快捷键绑定、多套互不相关的坐标命中检测、UI 提示逻辑等不同职责 → 记录为职责错位。

### 2.2 绘制/渲染层职责过载
- 一个绘制类根据不同列/角色/类型分支，绘制多种完全不同的视觉元素，且每个分支有独立坐标计算 → 记录为渲染职责过载。
- 绘制坐标常数与事件处理里的命中检测坐标分散在两处、各自硬编码、互不同步 → 记录为坐标耦合缺陷。
- 容器分配的空间与内容实际所需空间不一致（如硬编码尺寸前后矛盾）→ 记录为布局参数不一致。

### 2.3 数据层与业务层混杂
- 数据模型类内部直接执行磁盘 I/O、加密算法、手写线程调度 → 记录为数据层越权。
- 数据层直接触发 UI 反馈（弹窗、提示）而非通过信号/返回值交由上层处理 → 记录为层级倒挂。

### 2.4 反向依赖 / 越级耦合
- 子组件通过向上遍历 parent 链获取父级业务状态或调用父级方法 → 记录为反向依赖。
- 同一业务判断逻辑在多处以不同写法重复实现，缺少单一权威来源 → 记录为逻辑重复漂移。

### 2.5 历史"伪重构"识别
排查时需额外核查代码中标注"已重构"的位置（如带日期注释、commit 说明等），核实是否满足以下"有效重构"标准：**重构前一句话说不清楚职责的类/函数，重构后能否用一句话说清楚？** 不能，则该历史重构无效，必须在文档中标记为"历史重构未达标，问题实质仍存在"，不能因为存在重构注释就跳过排查。

---

## 3. 排查执行流程

1. **全量扫描**：逐文件列出所有类/主要函数，按第 2 节标准逐条核对。
2. **量化证据**：每条违规必须给出具体文件路径、类名/函数名、代码行为引用，禁止笼统描述"这里比较乱"。
3. **职责清单化**：对每个违规项，列出该类/函数当前实际承担的**全部**职责（不是举一个例子，是穷举）。
4. **拆分建议**：给出具体方案——新建什么类、负责什么单一职责、原类职责如何收敛、依赖关系如何解耦，不允许只写"应该拆分"。
5. **写入文档**：将以上内容按第 4 节格式追加进审查文档，不修改源代码。

---

## 4. 审查文档格式（持久化产出物）

在仓库中维护一份 `ARCHITECTURE_DEBT.md`（或指定路径），每条记录使用统一格式，按发现顺序追加，不覆盖已有记录：

```
## [编号] <文件路径> :: <类名/函数名>

- 状态：待处理 / 处理中 / 已解决
- 判定类型：<对应第2节标准编号，如 2.1 / 2.3>
- 发现日期：<日期>
- 职责清单（穷举当前承担的所有职责）：
  1. ...
  2. ...
  3. ...
- 代码证据：<具体引用，函数名+行为描述>
- 拆分方案：
  - 新建 <类名>：负责 <单一职责>
  - 原类收敛为：<单一职责>
  - 依赖解耦方式：<接口 / 信号槽 / 依赖注入等>
- 历史重构备注：<若此前有相关重构记录，注明是否有效；若无历史记录则留空>
- 优先级：高 / 中 / 低（依据：影响范围 × 修改成本）
```

要求：
- 每个违规项独立编号，不合并多条到一条记录里。
- "状态"字段初始一律为"待处理"，只有在后续确认代码已实际拆分完成后才可更新为"已解决"，且更新时需注明解决方式与解决日期。
- 文档需按"优先级：高"排在最前，方便后续按顺序处理。

---

## 5. 验收标准

排查任务是否合格，只看两条：

1. **代码库中是否还存在未被记录进文档、但明显命中第 2 节任一标准的类/函数？** 存在即排查不完整。
2. **文档中每一条记录是否都能回答"这个类/函数现在在做几件不相关的事"？** 答不出具体职责清单的记录，视为记录不合格，需补全。

不接受"大致排查了一遍""问题不算严重就没记"之类的模糊结项理由。
