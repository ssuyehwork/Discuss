# 大规模导入任务中的并发采样与 IO 优化 —— Analysis_Modification_Plan-85.md

## 1. 任务背景
在 ArcMeta 执行“扫描入库”或“文件夹导入”时，随着项目数量增加（特别是超过 10,000 项时），现有的串行处理模式会导致明显的 IO 瓶颈。虽然有后台进度条，但颜色解析算法的密集计算与磁盘随机读取分布不均，导致扫描周期过长且 UI 反馈存在迟滞感。

## 2. 问题定位

### 2.1 根因分析
1.  **串行递归**：`ImportHelper` 内部使用 `processPath` 递归函数。在一个任务中，文件是一个接一个被处理并提取元数据的。
2.  **频繁事务开销**：虽然每 500 项提交一次事务，但对于 SSD 而言，单线程的随机读取无法发挥带宽优势。
3.  **解析算法阻塞**：颜色解析在高分辨率图像上耗时较长，在大规模扫描时，这种延迟被线性放大。

### 2.2 涉及文件
- `src/util/ImportHelper.cpp`：导入调度器。
- `src/ui/BatchProgressDialog.h`：进度展示组件。

## 4. 详细解决方案

### 4.1 生产者-消费者导入模型
重构 `ImportHelper`，将导入任务分为三个阶段：

1.  **扫描器 (Scanner)**：单线程快速递归目录树，仅采集路径和基本属性，生成待处理队列（Queue）。
2.  **处理器集群 (Processors)**：使用 `QtConcurrent` 或自定义线程池启动多个工作线程。每个处理器从队列获取任务，执行：
    - 数据库注册（ManagedRole 设置）。
    - **低分辨率采样解析**：对于图像，优先执行极速预扫描（如 64x64 缩略图），快速获取近似主色调。
3.  **提交器 (Committer)**：主线程或专用落盘线程负责分批执行 SQL 写入。

### 4.2 动态 UI 降频技术
在 `BatchProgressDialog` 中引入“视觉缓冲”：
- 导入速度极快时（如 >1000 items/sec），不再为每个文件发送 `updateProgress` 信号。
- 改为固定频率刷新（如 60Hz 或 100ms），仅展示当前已完成的总数和最新的文件名，极大地释放主线程消息循环压力。

## 6. 实现准则与预警【核心】

1.  **并发冲突**：多个线程同时操作 SQLite 可能会触发 `SQLITE_BUSY`。必须确保所有写操作在一个独占事务中串行化，或者启用 WAL 模式并严格控制并发事务深度。
2.  **内存占用**：待处理队列应设有上限（如 5000 项），防止在包含百万文件的任务中导致内存溢出。
3.  **IO 优先级**：在后台大规模扫描时，应降低线程优先级（`QThread::LowestPriority`），确保不影响用户的前台实时操作体验。
