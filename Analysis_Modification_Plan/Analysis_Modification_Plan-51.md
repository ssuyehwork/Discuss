# 架构分析与修改方案 - 批量重命名交互增强（自动保存、配置导入导出及快速删除预设）

## 1. 现状剖析与缺陷诊断
当前的 `BatchRenameDialog` 在交互体验上存在三个显著的“物理延迟”：
- **手动保存负担**：用户修改规则后，若不手动存储，下次开启时将丢失设定，缺乏现代软件的“即改即存”特性。
- **预设系统封闭**：预设仅能保存在本地注册表/配置文件中，无法作为独立文件导出分享或备份。
- **预设删除路径长**：删除当前预设需要移动鼠标到右侧独立按钮，路径冗余，不符合“近场操作”原则。

---

## 2. 解决方案建议（已达成共识）

### 2.1 新文件名规则区：自动持久化 (Auto-Persistence)
**目标**：实现用户操作的无感记忆。
- **架构方案**：在 `BatchRenameDialog` 中引入**防抖持久化 (Debounced Persistence)** 机制。
- **逻辑流**：
    1. 监听 `RuleRow::changed` 信号及行增减操作。
    2. 触发 500ms 延迟定时器。
    3. 序列化当前的 `std::vector<RenameRule>` 为 JSON 字符串（存储类型、文本、起始值、补零位等）。
    4. 通过 `AppConfig::instance().setValue("LastBatchRenameRules", json)` 写入本地存储。
- **恢复逻辑**：窗口构造时，优先读取该配置并还原 `RuleRow` 序列。

### 2.2 预设操作按钮：重构为导入/导出 (Import/Export)
**目标**：打破配置孤岛，支持文件级交互。
- **改动点**：
    - 将 `m_btnSavePreset`（存储）更名为 **“导入...”**。
    - 将 `m_btnDeletePreset`（删除）更名为 **“导出...”**。
- **实现方案**：
    - **导出**：调用 `FramelessFileDialog::getSaveFileName`，将当前规则 JSON 写入 `.json` 文件。
    - **导入**：调用 `FramelessFileDialog::getOpenFileName`，读取 JSON 并通过 `onAddRow` 动态重建 UI 规则行。

### 2.3 预设下拉框：新增快速删除键 (Quick Delete "×")
**目标**：在选择点直接清理冗余配置。
- **UI 方案**：
    - **组合控件化**：将预设 `QComboBox` 与一个微型红色“×”按钮 (`QPushButton`) 封装在同一水平布局中。
    - **样式设计**：按钮尺寸设定为 20x20，圆角 4px，平时显示为灰色，悬停显示为 `#e81123` (红色)。
- **交互逻辑**：点击“×”时，直接从 ComboBox 的 Model 中移除当前项，并同步更新本地预设库。

## 3. 技术实现细节

### 3.1 规则序列化规范 (JSON)
```json
[
  {"type": "Text", "value": "Project_A"},
  {"type": "Date", "value": "yyyy-MM-dd"},
  {"type": "Sequence", "start": 1, "padding": 3}
]
```

### 3.2 UI 布局调整说明
- **预设区**：布局从 `[ComboBox] [存储] [删除]` 调整为 `[ComboBox][×] [导入] [导出]`。
- **间距规范**：`ComboBox` 与 `×` 之间设为 `2px` 的物理贴边感，`×` 与 `导入` 之间设为标准 `8px` 间距。

---

## 4. 结论
通过“自动记忆+开放式配置+近场清理”的三维优化，批量重命名的交互效率将从“工具级”提升至“专业级”，彻底消除用户在重复任务中的配置焦虑。
