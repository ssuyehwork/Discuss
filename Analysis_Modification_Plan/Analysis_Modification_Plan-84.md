# USN 实时同步中的 FID 物理对账优化 —— Analysis_Modification_Plan-84.md

## 1. 任务背景
目前 ArcMeta 的 `UsnWatcher` 与 `MftReader` 在处理文件变动（如重命名、移动）时，过度依赖路径字符串匹配。这种“基于路径”的对账模式在发生同名覆盖、快速移动或复杂目录重组时，极易产生元数据“误嫁接”——即新文件的物理身份被系统错误地赋予了旧文件的元数据逻辑，导致数据污染。

## 2. 问题定位

### 2.1 根因分析
1.  **身份识别单一**：系统主要使用路径作为唯一键，忽略了 NTFS 文件系统原生提供的 64-bit/128-bit File Reference Number (FRN/FID)。
2.  **覆盖漏洞**：当用户删除 A 文件并瞬间在同位置创建同名 B 文件时，若 USN 解析不够严谨，系统会认为 A 只是被“修改”了，从而将 A 的置顶、评分等元数据保留给 B。

### 2.2 涉及文件
- `src/mft/UsnWatcher.cpp`：USN 日志流解析逻辑。
- `src/mft/MftReader.h/cpp`：内存 SoA (Structure of Arrays) 的维护与检索。

## 4. 详细解决方案

### 4.1 强化 UsnWatcher 解析
修改 `UsnWatcher::run()` 与 `handleRecord`，确保从 USN 记录中提取 `FileReferenceNumber`。

1.  **V2/V3 适配**：对于 USN_RECORD_V2 使用 `FileReferenceNumber`，对于 V3 使用 `*reinterpret_cast<uint64_t*>(&v3->FileReferenceNumber)`。
2.  **对账链路**：在将变动通知 `MftReader` 时，必须携带该 FRN。

### 4.2 MftReader 物理对账算法
在 `MftReader::updateEntryFromUsn` 中引入以下逻辑：

1.  **路径反查**：根据 USN 中的路径，尝试从内存索引中找出旧条目。
2.  **FRN 校验**：
    - 若旧条目的 FRN 与新记录的 FRN **一致**：确认为重命名或移动，安全更新路径，保留元数据。
    - 若旧条目 FRN 与新记录 **不一致**：确认发生了“覆盖”或“偷梁换柱”。系统应将旧条目视为已删除，并为新 FRN 创建全新的数据记录，从而物理切断元数据污染。

## 6. 实现准则与预警【核心】

1.  **卷序列号对齐**：FRN 仅在单卷内唯一。在多卷监控环境下，对账逻辑必须结合卷序列号（Volume Serial）进行复合匹配。
2.  **性能平衡**：基于 FRN 的双向索引（Path -> FRN 和 FRN -> Path）会增加内存开销。需评估哈希表规模，对超大型卷（百万级文件）需优化内存占用。
3.  **日志风暴**：USN 日志在高 IO 下极其密集，对账逻辑必须保证 $O(1)$ 或 $O(\log N)$ 时间复杂度，严禁在 `handleRecord` 中执行磁盘读取。
