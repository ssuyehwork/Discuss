# 用户需求记录 - 失效数据监督机制

## 需求背景
ArcMeta 需要新增一套“失效数据”监督机制，用于检测托管库（ArcMeta.Library_[盘符]）内的文件/文件夹是否被【第三方应用】（Windows 资源管理器、其他软件等，非 ArcMeta 自身）删除。

## 核心规则
1.  **判定“失效”的唯一触发条件**：托管库范围内的文件/文件夹被外部应用真实删除（NTFS USN Journal 产生的非库内移动类删除事件）。
2.  **硬删除路径**：若用户在 ArcMeta 应用内部主动执行“将库内文件/文件夹移动到库外其他目录”，直接从数据库抹除记录。
3.  **操作分流判断**：复用 `internalOperating` 状态：
    - `internalOperating == true` 期间的移除事件：视为应用内操作，执行硬删除。
    - `internalOperating == false` 期间的移除事件：视为第三方外部删除，标记为失效（`is_invalid = 1`）。
4.  **UI 行为**：用户点击“失效数据”分类节点时，左侧“分类”、中间“目录导航”、右侧“筛选”面板应收敛隐藏，仅展示扁平化的失效数据列表。

## 技术确认
- **失效状态存储**：复用 `metadata` 表中的 `is_invalid` 字段。
- **重命名级联**：由于 `metadata` 存储完整绝对路径，更名操作需实现批量路径前缀替换的级联逻辑。

## 附加优化 (Bug Fix)
- 修复 `renameItem` 中扩展名倒排索引指向错误容器的 Bug。
- 为 `renameItem` 引入批量事务保护（内存库与磁盘库同步任务），防止大规模移动时性能假死。
- 将 `renameItem` 逻辑异步化（QtConcurrent::run），杜绝主线程阻塞。
