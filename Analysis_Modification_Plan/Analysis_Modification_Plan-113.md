# ArcMeta 核心架构重构方案 (Analysis_Modification_Plan-113)

本方案旨在彻底重构 ArcMeta 的数据入库、分类管理及物理同步逻辑，确立以“托管库”为核心的物理-逻辑镜像体系，解决现有实现中的性能抖动与逻辑冗余。

## 1. 核心需求对齐 (Requirement Alignment)

*   **入库状态治理**：解决“进度弧线（百分比圆圈）”针对“所有文件夹”绘制导致的 I/O 浪费（用户原话：“进度弧线（百分比圆圈）：如果该数值处于 0.0 到 1.0 之间，代码会调用 painter->drawArc 绘制一个蓝色的进度弧。”）。
*   **物理分类一体化**：确立 `ArcMeta.Library_盘符` 为唯一数据管理入口，实现物理路径与逻辑分类的深度集成。
*   **冗余清理**：废除无意义的虚拟分类项（用户原话：“既然通过‘ArcMeta.Library_盘符’统一管理数据，那么侧边栏分类中的‘未分类’纯属冗余的了，所以可以将其移除”）。

## 2. 详细设计方案 (Detailed Design)

### 2.1 存储与登记架构 (Storage & Status Machine)
*   **废除临时表**：彻底废除 `pending_imports` 表，将 USN 监听到的变动直接标记在核心 `metadata` 表中。
*   **状态机引入与视觉映射**：
    *   `Status::Registered (0)`：已登记，指纹未解析。UI 显示**蓝色进度弧**（数值基于 SQL 实时聚合子项状态产生，杜绝物理扫描）。
    *   `Status::Ingested (1)`：已入库，元数据全量就绪。UI 显示**绿色对勾**。
    *   `Status::Invalid (-1)`：已失效（迁出库外）。UI 显示为半透明灰度。

### 2.2 物理准入与迁移逻辑 (Physical Access & Migration)
*   **托管库强绑定**：仅允许位于 `ArcMeta.Library_盘符` 路径下的项目进入 `Ingested` 状态。
*   **自愈创建 (Auto-Creation)**：执行导入操作时，若根目录缺失托管库文件夹，系统自动静默创建。
*   **迁移即激活**：库外项目执行“扫描入库”或“拖拽”时，系统强制执行同盘 `Move` 操作至托管库，随后触发异步解析。

### 2.3 失效数据管理 (Invalid Data Management)
*   **失效不删除**：从托管库迁出的项目标记为 `Invalid`。
*   **失效视图**：侧边栏保留“失效数据”入口。列表模式下，失效项图标采用**半透明灰度处理**，路径显示为“最后已知物理位置”。
*   **手动清理与复活**：支持 Checkbox 批量物理清除元数据；若项目重新移回托管库，基于 FRN 自动复活元数据。

### 2.4 跨库迁移与身份识别 (Cross-Library & Heuristic)
*   **FRN 追踪**：同磁盘迁移基于 FRN 实时更新路径，维持“已入库”状态。
*   **启发式匹配与冲突解决**：
    *   跨磁盘迁移导致 FRN 改变时，通过“文件名 + 大小 + 修改时间”进行启发式重识别。
    *   **重复添加处理**：若检测到完全匹配的冲突，调用 `ConflictResolutionDialog` 提示用户。支持“使用已存在项目（复用元数据）”或“保留两者（分流存储）”模式。

### 2.5 侧边栏与同步逻辑 (Sidebar & Sync)
*   **一等公民地位**：托管库物理文件夹作为“主分类”常驻侧边栏，支持“快速访问”固定。
*   **双向重命名同步**：
    *   **逻辑 -> 物理**：侧边栏改名自动触发磁盘 `Rename`。
    *   **物理 -> 逻辑**：USN 感知到库内文件夹改名，执行“成功即同步”逻辑。通过 Windows 文件系统事件（如 `FILE_ACTION_RENAMED_NEW_NAME`）确认 IO 成功后再更新逻辑分类名。
*   **冗余清理**：从 `CategoryModel`、右键菜单（回归“未分类”）中彻底移除“未分类”相关代码。

## 3. 影响评估 (Impact Assessment)

*   **性能提升**：废除盲目的全量文件夹递归扫描，UI 刷新由数据库字段状态直接驱动，消除 I/O 抖动。
*   **交互简化**：用户不再需要区分“物理文件夹”和“逻辑分类”，托管库即分类，操作即同步。
*   **数据安全**：失效数据标记机制确保了用户元数据（评分、标签）在误操作或磁盘调整时不被意外抹除。

## 4. 风险控制与兜底 (Risk Control)

*   **同步原子性**：针对物理重命名，系统采用“事件后触发”机制。若物理重命名失败（如被占用），由于底层未产生 USN 变更信号，逻辑层自动保持原样，无需显式回滚。
*   **数据隔离**：托管库内外的元数据状态由 `ingestion_status` 严格隔离，杜绝逻辑污染。
