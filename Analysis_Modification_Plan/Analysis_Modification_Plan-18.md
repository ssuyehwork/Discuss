# Analysis_Modification_Plan-18: 高级筛选器“颜色标记”性能优化与交互重构

## 1. 现状逻辑分析 (Current Logic Analysis)

### 1.1 O(N) 渲染瓶颈
目前 `src/ui/FilterPanel.cpp` 中的 `rebuildGroups()` 函数采用了一种极其低效的渲染模式：
- **逻辑**：遍历 `m_colorCounts`（当前目录及其子目录下所有独特颜色的集合）。
- **行为**：对每一个色码（Hex String）调用 `addFilterRow`。
- **后果**：`addFilterRow` 会 `new` 出一个包含 `StyledCheckBox`、`QLabel`（色点）、`QLabel`（标签）、`QLabel`（计数）的复合容器。
- **灾难点**：如果用户处理的是包含丰富色彩（如摄影、设计稿）的文件库，独特颜色可能成千上万。Qt 在布局中管理成千上万个复合 Widget 会导致严重的：
    1. **内存占用**：控件树急剧膨胀。
    2. **计算负担**：布局引擎在重算大小时会产生指数级的开销。
    3. **视觉噪音**：无限长的复选框列表让用户根本无法有效操作。

### 1.2 交互局限性
- **精确匹配 vs. 感知筛选**：目前的筛选主要依赖 Hex 值的精确匹配或极简的 RGB 距离。用户无法通过点击一个“红色”来感知并覆盖所有“深红、粉红、大红”的文件。

---

## 2. 解决方案 (Proposed Solution)

### 2.1 引入“色块矩阵”交互 (Color Block Grid)
彻底废弃“一色码一复选框”的列表模式，在 `InlineHueSlider` 下方引入固定规格的色块展示区。

**布局设计：**
- **常用颜色区域 (Static Common Colors)**：
    - 预设 12-16 个标准物理色块（红、橙、黄、绿、青、蓝、紫、粉、黑、白、灰、棕）。
    - 规格：24x24px 圆角矩形，流式布局（Flow Layout）。
- **最近筛选区域 (Dynamic Recent Colors)**：
    - 展示用户最近筛选过的 8 个色块。
    - 采用 **LRU 缓存机制**：每次用户从色相杆提取颜色或使用拾色器选择颜色后，该颜色进入缓存头部。

### 2.2 近色筛选算法 (Near-color Filtering)
引入基于色彩感知的匹配逻辑，而非简单的字符串匹配。

**技术实现方案：**
1. **色彩空间转换**：将颜色从 RGB 转换为 **CIELAB (L*a*b*)** 空间。LAB 空间在人类感知上是均匀的，更适合计算“看起来像不像”。
2. **Delta E 计算**：当点击某个色块时，计算目标颜色与文件中颜色之间的 **Delta E (Euclidean distance in LAB)**。
3. **模糊匹配引擎**：
    - 默认容差阈值设定为 `10.0`（人眼感知的“显著相似”）。
    - 内容面板过滤逻辑改为：`distance(file_color, target_color) < threshold`。

### 2.3 性能加固 (Performance Hardening)
- **静态化控件**：色块矩阵在 `FilterPanel` 创建时一次性初始化，后续仅通过 `setProperty("count", n)` 或 `update()` 刷新。
- **异步对账**：颜色计数的统计（即“该色块对应多少个文件”）在后台线程计算，通过信号通知 UI 更新数字，杜绝阻塞主线程。

---

## 3. 架构演进标准

1. **固定渲染消耗**：无论数据库里有多少色码，UI 控件数量保持恒定。
2. **直观感性筛选**：用户点击标准色块即可完成大类筛选。
3. **自动学习交互**：动态色块区域自动沉淀用户高频使用的配色方案。
