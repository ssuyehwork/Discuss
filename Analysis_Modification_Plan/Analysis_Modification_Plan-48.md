# 架构分析与修改方案 - 批量重命名功能边界扩展（解除文件类型限制，实现全类型支持）

## 1. 现状剖析
在 `ContentPanel.cpp` 的右键菜单构建逻辑中，目前存在一个显式的条件分支（约 1422 行）：
```cpp
if (isFolder) { 
    menu.addAction("批量重命名 (Ctrl+Shift+R)")->setData(ActionBatchRename); 
} else { 
    // 显示加密保护菜单...
}
```
此逻辑导致“批量重命名”功能被物理锁定在“文件夹”对象上。当用户选中一个或多个文件时，由于 `isFolder` 为假，该功能入口被彻底隐藏，违背了用户期望全类型支持的直觉。

## 2. 解决方案建议（已达成共识）

### 2.1 逻辑解耦：从“类型属性”转向“操作意图”
批量重命名的本质是针对一组路径（Paths）的规则化处理。底层 `performBatchRename()` 已经具备处理 wstring 列表的能力，因此 UI 层面应解除对“文件夹”的强制依赖。

### 2.2 菜单显示逻辑优化方案
建议将右键菜单的准入条件修改为基于“多选状态”与“特定类型”的组合判定，以确保交互的合理性。

**修改建议：**
- **逻辑表达**：将判断条件修改为 `if (isFolder || selectedCount > 1)`。
- **交互逻辑描述**：
    1. **多选状态下**（无论文件还是文件夹）：强制显示“批量重命名”，作为首选的集体操作。
    2. **单选文件夹时**：保留“批量重命名”入口，允许用户对文件夹及其内部内容进行重构。
    3. **单选文件时**：默认保持现有的行内“重命名”体验，但底层快捷键 (Ctrl+Shift+R) 应保持可用。

### 2.3 快捷键全局一致性
需确保快捷键 `Ctrl+Shift+R` 在 `ContentPanel` 焦点内对所有选中项（不论文件还是文件夹）均能稳定触发 `performBatchRename()` 流程。

## 3. 预期改进效果
- **全类型覆盖**：用户可以对任意选中的文件、文件夹或混合集合执行批量重命名。
- **交互直觉化**：修复了“明明选了多个文件却找不到批量处理入口”的逻辑偏差，提升了工具的生产力属性。

## 4. 结论
通过解除 `isFolder` 的硬编码限制，使批量重命名功能回归其作为“路径处理器”的本质，实现对文件与文件夹的无差别支持，确保 UI 交互逻辑与底层功能能力完全对等。
