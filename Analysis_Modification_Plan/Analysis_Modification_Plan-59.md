# 递归逻辑架构重构：卷级统一库与 USN 增量驱动 —— Analysis_Modification_Plan-59.md

## 1. 任务背景
目前递归模式为每个文件夹创建独立的 Scoped DB，导致资源利用率低下且缺乏有效的增量同步。用户要求重构为“一硬盘一库”模式（`Recursion_{VolumeSerial}.db`），并引入按需驱动的 USN Journal 同步逻辑。

## 2. 架构开发红线【核心准则】
应用底层严格划分为两种互不干扰的驱动模式。**本次设计的“递归”功能必须严格锁定在“磁盘路径模式”下，严禁与“分类模式”混淆。**

| 模式 | 驱动方式 | 作用域 |
|------|----------|--------|
| **侧边栏分类模式** | DB 驱动 | 当前选中分类及其逻辑子类（数据库查询） |
| **磁盘路径模式** | 实时 I/O 驱动 | 当前物理路径及其子目录（实时扫描） |

> **红线说明**：递归行为是物理路径扫描能力的扩展。`Recursion_*.db` 仅作为物理路径模式下的加速镜像（缓存），其数据有效性必须由物理磁盘状态（通过 USN/I/O）单向驱动。

## 3. 问题定位
*   **模块**：`DatabaseManager` (引用计数与句柄管理), `MetadataManager` (物理卷 ID 获取), `RecursionSyncEngine` (按需同步控制), `ContentPanel` (UI 状态驱动)。
*   **核心痛点**：需确保递归模式在物理路径作用域内运行，并实现磁盘卷级库的精准生命周期管理。

## 5. 详细解决方案

### 5.1 卷驱动的库加载机制 (DatabaseManager)
*   **物理卷识别**：通过 `MetadataManager::getVolumeSerialNumber` 获取路径所属磁盘的序列号。
*   **引用计数管理**：
    - 在 `DatabaseManager` 维护 `m_recursionRefCount`。
    - **加载**：计数从 0 变 1 时，物理打开 `Recursion_{volSerial}.db` 并建立内存镜像。
    - **卸载**：当计数归 0 时，执行 `sqlite3_backup` 刷盘并物理关闭磁盘与内存库句柄，回收所有内存页。

### 5.2 按需 USN 增量驱动 (RecursionSyncEngine)
*   **实时同步逻辑**：
    - 当卷库加载后，同步引擎激活对应卷的 `UsnWatcher`。
    - 该引擎仅负责将物理磁盘的变动（USN 日志）单向同步至 `Recursion_*.db` 镜像。
    - **物理优先**：任何时候以物理磁盘状态为准。

### 5.3 递归触发器重构 (ContentPanel)
*   **模式解耦**：
    - 确保 `dataSourceChanged("nav")` 状态下才激活递归数据库。
    - 退出递归或切换到 `dataSourceChanged("category")` 时，必须触发对应卷库的卸载。

### 5.4 内存与性能红线
*   **卸载即回收**：必须调用 `sqlite3_close`，严禁无引用的库驻留内存。
