# 置顶功能排序逻辑失效与上下文冲突分析 —— Analysis_Modification_Plan-74.md

## 1. 任务背景
用户反馈在对文件夹、子分类或文件执行“置顶”操作后，项目位置未发生预期变动（未排至区块最前方）。需分析是否存在逻辑架构冲突、角色覆盖或数据刷新机制漏洞。

## 2. 逻辑架构审计结论

### 2.1 权重等级冲突 (Weight Hegemony)
在 `FilterProxyModel::lessThan` 函数中，排序逻辑被划分为严格的阶梯层级：
1. **类型层 (Type Layer)**: `leftType == "folder" || leftType == "category"` 具有绝对最高优先级。
2. **状态层 (State Layer)**: `leftPinned` 判定处于第二优先级。

**冲突表现**：
系统强制执行“文件夹置顶于文件”的物理策略，导致**置顶的文件权重低于未置顶的文件夹**。在混合视图（如截图所示）中，文件仅能在“文件集合”内部置顶，无法跨越类型边界。

### 2.2 子分类元数据盲区 (Category Data Void)
在 `FerrexVirtualDbModel::data` 的实现中：
* **常规分支**: 响应 `PinnedRole` 并返回 `record.pinned`。
* **分类分支 (`record.isCategory`)**: 漏掉了对 `PinnedRole` 和 `IsLockedRole` 的处理，默认返回空。

**影响**：排序引擎在对比子分类时，由于获取到的置顶角色始终为 `false`，导致置顶操作对子分类的物理排序序列完全无效。

### 2.3 角色一致性冲突 (Role Inconsistency)
系统中存在 `PinnedRole` (Role 101) 与 `IsLockedRole` (Role 102) 的功能重叠：
* **UI 渲染 (GridItemDelegate)**: 依赖 `IsLockedRole` 绘制图钉图标。
* **右键菜单 (ActionPin)**: 通过 `setData` 修改 `IsLockedRole`。
* **排序引擎 (FilterProxyModel)**: 优先判定 `PinnedRole`，若有效则不再看 `IsLockedRole`。

**数据流断裂**：
在 `setData` 执行后，如果模型仅触发了 `IsLockedRole` 的 `dataChanged`，而 `PinnedRole` 由于布尔值 `false` 被 ProxyModel 视为“有效但无变化”，则可能导致动态排序（Dynamic Sort）不被触发。

### 2.4 持久化逻辑失效 (Persistence Failure)
* **现状**: `setData` 仅调用 `MetadataManager::setPinned`，这依赖于物理路径。
* **缺陷**: 子分类（Category）在模型中 `path` 为空。这意味着置顶分类时，不仅元数据没存进去，连内存记录的 `updateRecordMetadata("")` 也会因找不到 Key 而失败。

## 3. 工业级解决方案建议 (方案对比)

### 方案一：绝对权重提升 (不推荐)
修改 `FilterProxyModel::lessThan`，将置顶对比提升至全局第一优先级。
*   **优点**: 置顶项绝对排在第一位。
*   **缺点**: 置顶文件会排在文件夹上方，破坏“结构优先”的视觉习惯，导致布局杂乱。

### 方案二：双区隐式划分 (强烈推荐 - 架构共识)
逻辑上将内容区划分为“结构区（文件夹/分类）”与“内容区（文件）”。置顶操作仅在各自所属区域内生效。
*   **逻辑实现**:
    1.  **第一权重**: 保持 `leftIsDir != rightIsDir` 判定，确立隐式分界。
    2.  **第二权重 (区内置顶)**: 在同类型项目对比中，引入置顶判定。
*   **优点**: 既保留了文件夹置顶显示的物理层级，又解决了“置顶文件被普通文件夹掩埋”的逻辑矛盾。

### C. 补全分类模型接口
在 `FerrexVirtualDbModel::data` 的 `isCategory` 分支中增加：
```cpp
else if (role == IsLockedRole || role == PinnedRole) {
    return record.pinned;
}
```

### C. 强化动作执行链 (Action Chain Hardening)
在 `ContentPanel::onCustomContextMenuRequested` 处理 `ActionPin` 时：
1. 识别选中项是否为分类，若是，则调用 `CategoryRepo::update` 进行持久化。
2. 在 `setData` 完成后，显式调用 `m_proxyModel->invalidate()` 以强制全视图物理重排。

### D. 显式排序激活 (Sort Activation)
在 `loadCategory` 或 `loadPaths` 数据填充完毕后，显式执行 `m_proxyModel->sort(0, Qt::AscendingOrder)`，确保视图从“插入序”切换至“逻辑序”。

## 4. 风险评估
* **性能影响**: 提升置顶权重会增加 `lessThan` 的调用频率，但在虚拟化模型下（百万级条目）性能损耗微秒级，可忽略。
* **布局抖动**: 强制 `invalidate()` 可能导致大目录下滚动条位置微调，需结合 `QPersistentModelIndex` 保持焦点。

## 5. 结论确认
项目置顶失效并非单一 Bug，而是**类型权重覆盖（主要原因）**与**分类数据链路断裂（次要原因）**共同导致的架构性覆盖。执行上述重构可彻底修复该问题。
