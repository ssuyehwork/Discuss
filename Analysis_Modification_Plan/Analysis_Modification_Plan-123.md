# DatabaseManager 读写分离锁高性能优化 —— Analysis_Modification_Plan-123.md

## 1. 任务背景
在 Plan-122 的并发审计中发现，`DatabaseManager` 使用单一的 `std::mutex` 保护所有操作。这导致当后台 I/O 线程正在执行长时间的同步或数据库加载时，主线程（UI 线程）若调用 `getMemoryDb` 获取句柄，会被迫挂起等待，产生微小但可感知的卡顿。为了追求极致丝滑，需将锁机制升级为读写分离锁。

## 2. 问题定位
- **瓶颈函数**：`getMemoryDb`、`getDiskDb`、`getGlobalDb`。这些函数仅执行 `map` 查找，属于典型的“读操作”。
- **竞争源**：`init`、`shutdown`、`loadDb`、`closeDb`。这些函数修改容器或句柄状态，属于“写操作”。
- **风险点**：单一互斥锁导致读操作无法并发，且必须排队等待写操作完成。

## 4. 详细解决方案

### 4.1 锁类型升级 (Header 修改)
- **修改点**：在 `DatabaseManager.h` 中：
    1.  包含头文件 `<shared_mutex>`。
    2.  将 `std::mutex m_mutex;` 更名为 `std::shared_mutex m_dbMutex;`，以清晰表达其读写分离语义。

### 4.2 查询接口优化 (共享读锁)
- **目标函数**：`getMemoryDb`、`getDiskDb`、`getGlobalDb`。
- **实施细节**：
    - 使用 `std::shared_lock<std::shared_mutex> lock(m_dbMutex);`。
    - 这允许多个线程同时进入这些函数进行句柄查询，只要没有写锁正在持有。

### 4.3 维护接口优化 (排他写锁)
- **目标函数**：`init`、`shutdown`、`loadDb`、`closeDb`、`getMemoryDb`（涉及重命名重开的部分）。
- **实施细节**：
    - 使用 `std::unique_lock<std::shared_mutex> lock(m_dbMutex);`。
    - 确保在修改 `m_driveDbs` 映射或关闭数据库连接时，没有其他读写操作正在进行。

### 4.4 针对“盘符漂移”逻辑的特殊处理
- **逻辑重构**：在 `getMemoryDb` 中，原本包含“检测到盘符变化执行重命名并重新打开”的逻辑（涉及写操作）。
- **策略**：
    1.  先以 `shared_lock` 进入，检查是否需要迁移。
    2.  若不需要迁移（99% 的情况），直接返回句柄并释放读锁。
    3.  若检测到需要迁移，释放读锁，升级/重新获取 `unique_lock` 执行迁移操作。这种“双重检查”模式可极大提升常规性能。

## 6. 实现准则与预警【核心】
- **编译器支持**：`std::shared_mutex` 需要 C++17 支持。若环境受限，需降级使用 `std::shared_timed_mutex` (C++14) 或 Qt 的 `QReadWriteLock`。**经审计，本项目已使用 C++17。**
- **死锁预防**：严禁在持有 `unique_lock` 的情况下再次申请自身的锁（即便读写分离锁也不支持重入）。
