# 筛选器性能瓶颈重构与 UI 复用方案 —— Analysis_Modification_Plan-89.md

## 1. 任务背景
在完成性能审计后，针对用户反馈的“搜索框与筛选逻辑是否独立”及“筛选操作延迟”问题，本方案深入分析了搜索结果呈现与筛选面板刷新之间的耦合机制，并提出针对性的重构计划。

## 2. 问题定位
- **渲染瓶颈**：`FilterPanel::rebuildGroups()` 在每次统计更新时执行 `deleteLater()` 销毁数百个控件并从零重建，造成主线程重度绘图负载。
- **计算瓶颈**：`FilterProxyModel::filterAcceptsRow` 在每一行显示时同步执行 `calculateDeltaE`（包含大量浮点运算），导致 CPU 密集型任务阻塞事件循环。
- **耦合瓶颈**：搜索框输入触发模型更新，进而通过信号链强制 `FilterPanel` 全量重绘，形成性能瓶颈连带效应。

## 4. 详细解决方案

### 4.1 架构独立性与耦合点分析
- **逻辑独立性**：搜索框（Search）负责改变数据源的关键词匹配，筛选器（Filter）负责视图层的布尔判定。
- **物理耦合点**：
    1. `FilterProxyModel` 顺序执行：先跑筛选器判定，再跑搜索词判定。
    2. 信号链耦合：搜索结果流式回传会导致 `recalculateAndEmitStats` 高频触发，进而引爆 `FilterPanel` 的重建风暴。

### 4.2 FilterPanel：从“重建”转向“更新” (UI Pool)
**重构路径：**
1. **输入框保护**：将搜索/过滤输入框从 `rebuildGroups` 的清理范围中剔除，固定在 GroupHeader 之后。
2. **复选框池化**：
   - 维护 `QList<ClickableRow*> m_rowPool`。
   - 更新时，仅修改现有控件的 `setText()` 和计数，避免 `deleteLater()` 带来的主线程 GC 压力。

### 4.3 FilterProxyModel：算法减负
**优化逻辑：**
1. **预计算缓存**：在 `ItemRecord` 中增加 Lab 空间颜色缓存。
2. **快速判定**：`filterAcceptsRow` 仅执行欧氏距离平方和计算，剔除 `pow` 和 `sqrt` 调用。

## 6. 实现准则与预警【核心】
1. **UI 抖动预警**：从销毁重建切换为复用时，需确保护理好 Layout 的伸缩性，防止出现控件闪烁或跳动。
2. **数学精度预警**：优化 `calculateDeltaE` 时，阈值比较必须统一使用平方空间，防止判定逻辑失效。
