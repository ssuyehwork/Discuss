# 架构分析与修改方案 - 批量重命名全场景解锁（消除磁盘/数据库视图判定差异）

## 1. 现状剖析与缺陷诊断

### 1.1 问题描述
目前“批量重命名”功能在 UI 呈现上存在严重的“来源歧视”与“类型歧视”：
- **来源歧视**：在物理磁盘路径视图下工作正常，但在数据库逻辑视图（如“全部数据”、“标签管理”）下，即使选中了多个文件，右键菜单中也无法看到该选项。
- **类型歧视**：菜单显示逻辑被硬编码限制在“文件夹”对象上。

### 1.2 逻辑根因
审计 `src/ui/ContentPanel.cpp` 发现，右键菜单的显示逻辑完全依赖于 `isFolder` 变量：
```cpp
bool isFolder = onItem && (currentIndex.data(TypeRole).toString() == "folder");
...
if (isFolder) { 
    menu.addAction("批量重命名 (Ctrl+Shift+R)")->setData(ActionBatchRename); 
}
```
在数据库视图中，即使选中了多个文件，由于这些项目的 `TypeRole` 是 `file` 而非 `folder`，导致条件判定永远失败。

---

## 2. 解决方案建议（已达成共识）

### 2.1 架构升级：从“类型驱动”转向“路径驱动”
批量重命名的核心能力是处理“一组有效的物理路径”。应彻底废除基于 `TypeRole` 的单一判定。

### 2.2 菜单准入逻辑重构方案
建议在 `onCustomContextMenuRequested` 中引入 **`selectedCount`**（选中项总数）判定，并结合物理路径验证：

**修改建议逻辑：**
1. 获取当前选中的有效项目总数 `selectedCount`。
2. **显示条件**：满足以下任一条件即解锁批量重命名：
   - 选中项 `selectedCount > 1`（不论是文件还是文件夹）。
   - 选中项为文件夹（即使只选了一个，也允许进入高级面板处理子项）。
3. **前置校验**：确保选中的路径在 `PathRole` 中有效存在且非只读/系统保护路径。

### 2.3 跨视图兼容性保障
由于 `performBatchRename()` 底层直接从模型中提取 `PathRole`，只要上述 UI 准入开关打开，数据库视图中的文件路径将能无缝传递给 `BatchRenameDialog`。

## 3. 规范要求
- **全场景一致性**：无论用户处于“磁盘目录”、“系统分类”还是“自定义标签”视图下，右键多个文件必须看到“批量重命名”。
- **交互逻辑**：
  - 单选文件：保持行内重命名。
  - 单选文件夹或多选任意项：显示“批量重命名”。

## 4. 结论
通过将右键菜单逻辑从死板的类型判断提升到“路径处理意图”的高度，可以实现全软件内批量处理能力的百分之百一致性，彻底消除数据库视图下的功能缺失感。
