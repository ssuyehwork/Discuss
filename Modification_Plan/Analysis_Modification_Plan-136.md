# Analysis_Modification_Plan-136: 拒绝盲目——工业级“无感”同步架构深度设计

本方案是对之前所有“修补式”方案的彻底否定，旨在从底层锁模型与 I/O 调度层面重建系统的性能基石。

## 1. 锁模型革命：从“全互斥”到“读写分离”

### 1.1 现状死穴
`MetadataManager::m_mutex` (shared_mutex) 导致写操作（后台同步）与读操作（UI 刷新/搜索）在内存层面硬碰硬。后台写 10ms，UI 就要等 10ms，造成微卡顿积累。

### 1.2 方案：RCU (Read-Copy-Update) 快照内存镜像
*   **实现细节**：
    *   将 `m_cache` 封装在 `std::shared_ptr<const CacheMap>` 中。
    *   **读操作 (UI)**：通过原子操作获取 `shared_ptr` 的副本，全程无锁读取快照。
    *   **写操作 (后台)**：
        1. 拷贝当前 `m_cache`。
        2. 在副本上修改。
        3. 利用 `std::atomic_store` 原子级替换全局指针。
*   **收益**：UI 线程的元数据读取耗时恒定为 $O(1)$，且绝对不会被后台事务挂起。

## 2. 数据库单轨制优化：双句柄隔离 (Dual Handle)

### 2.1 方案：UI 读写完全隔离
*   **UI 线程**：持有独立的 `sqlite3*` 句柄，仅执行 `SELECT`，并开启 `PRAGMA query_only = ON`。
*   **后台线程**：持有写句柄，开启 WAL 模式，执行 `INSERT/UPDATE`。
*   **收益**：利用 SQLite WAL 模式特性，实现“读不锁写，写不锁读”。UI 界面在大规模数据入库时依然能瞬间返回搜索结果。

## 3. I/O 调度：废除暴力对账，引入“USN 水库”

### 3.1 现状死穴
`syncAllManagedLibraries` 在洪流时发起的全盘扫描是磁盘的寻道自杀行为。

### 3.2 方案：USN 缓冲区 (Reservoir) 与延迟解析
*   **动作**：
    1. USN 信号进来后，只做最轻量级的 FID/Path 占位登记（$O(1)$ 写入）。
    2. 建立 `MetadataTaskQueue`，将“提取尺寸、颜色”等重型操作标记为“Pending”。
    3. **按需触发**：仅当用户滚动列表，项目进入可视区域 (Viewport) 时，由 UI 信号提升该任务优先级进行“即时解析”。
    4. 其余任务在 CPU/IO 空闲时后台静默处理。
*   **收益**：彻底消灭由于大批量导入导致的磁盘 IO 峰值，让系统资源曲线变平滑。

## 4. 彻底解决串行化：分库锁池 (Lock Pooling)

### 4.1 方案
在 `DatabaseManager` 中，废除全局 `m_mutex`。改为：
```cpp
std::map<std::wstring, std::unique_ptr<std::mutex>> m_dbLocks;
```
按驱动器序列号分配独立的互斥锁。
*   **收益**：多硬盘环境下，不同盘符的数据库加载与操作完全并行，互不干扰。

## 5. 结论
之前的方案（Plan-131~135）确实存在“头疼医头”的盲目性。**Plan-136** 是真正从计算机底层原理出发，解决读写竞争与磁盘寻道成本的终极方案。如果不动这根地基，其它的优化都只是在沙滩上盖楼。
