# 用户的设计理念和整体蓝图规划与架构 Overall Planning and Architecture.md

本文档记录 ArcMeta 项目的核心设计原则，是所有后续开发、重构、AI 辅助编码
（Jules 等）必须遵守的最高准则。无论是对现有代码的理解出现偏差，还是新增
功能时的方案选择，都不可违背本文档中记录的原则。如果发现某次改动与本文档
冲突，应当以本文档为准，改动方案需要重新调整，而不是绕开或悄悄修改本文档
的既定原则。

---

## 核心原则一：必须模块化、职责单一，禁止临时拼凑

**规则**：项目的任何模块、类、函数，都必须遵循单一职责原则（SRP）。当需要
新增功能或修复问题时，宁可承担重构的成本，也不允许为了"快速让功能跑起来"
而采取临时拼凑、职责混杂、模块间硬编码耦合的实现方式。

**理由**：临时拼凑式的实现，短期内看起来"能用"，但会在后期引发严重后果，
具体表现为：

- 一个模块的问题会意外牵连、拖慢另一个本应完全无关的模块（例如全局大锁
  导致不同磁盘分区的独立操作互相阻塞排队）
- 代码逻辑分散在不该出现的位置（例如 UI 控件类内部直接编写 SQL 语句、
  直接操作数据库连接），导致修改一处逻辑需要同时改动多个不相关的文件
- 相同的业务逻辑在多个模块中被重复实现，出现行为不一致、难以维护的情况
- 关键功能被硬编码锁死在不合理的实现细节上（例如标签功能被硬编码只读写
  C 盘分库，与项目"多分库、按盘符隔离"的整体设计矛盾）
- 问题难以定位和独立修复：架构混乱时，一个卡顿或崩溃现象往往牵扯多个
  模块互相纠缠，无法判断真正的根因；模块化清晰时，问题能被快速锁定在
  某一层的责任范围内

**具体要求**：

1. 数据访问逻辑（SQL 语句、prepared statement 管理、数据库连接获取）
   必须收拢到统一的 Repository / 持久层，业务逻辑层和 UI 层不允许直接
   编写或执行 SQL
2. UI 渲染逻辑、业务状态管理、数据库读写、文件系统操作、操作系统底层
   API 调用，应当分属不同的模块，不允许混杂在同一个类或同一个函数内
3. 模块之间的通信应当通过明确定义的接口（信号槽、公开方法）进行，
   禁止一个模块直接访问、依赖另一个模块的内部实现细节（例如通过遍历
   父级组件树、强制类型转换来调用另一个具体类的方法）
4. 禁止使用全局大锁笼统地保护多个互不相关的资源，导致原本可以并发执行
   的操作被迫串行等待
5. 遇到需要快速修复的紧急问题时，允许先用最小改动止血，但必须在后续
   安排正式的模块化重构，不允许把"临时方案"当作最终方案长期保留
