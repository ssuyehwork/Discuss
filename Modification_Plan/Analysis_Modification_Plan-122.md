# 并发安全性与 UI 丝滑度架构评估报告 —— Analysis_Modification_Plan-122.md

## 1. 任务背景
在实施了“内存优先实时增量同步”与“SyncStatusService 试点解耦”后，系统已具备运行期增量落盘与秒退出的基础。本报告旨在评估当前代码在极端并发场景下（如高频元数据修改、大批量标签重命名）是否仍存在潜在的“锁竞争、假死或卡顿”风险。

## 2. 审计对象
- `src/meta/DatabaseManager.cpp/h`：连接句柄管理与后台队列。
- `src/meta/MetadataManager.cpp/h`：增量同步分发链路。
- `src/core/SyncStatusService.cpp/h`：UI 状态节流器。

## 3. 核心评估结论

### 3.1 丝滑度达成度：优
- **SyncStatusService**：成功通过 200ms `QTimer` 节流将高频持久化信号收拢。UI 图标刷新频率被物理限制在合理范围，彻底杜绝了消息风暴。
- **异步分发**：磁盘 I/O 被成功剥离至后台 `workerThread`，主线程不再承载昂贵的物理写盘开销。

### 3.2 潜在风险点：中度 (需关注)
通过代码审计，发现以下可能导致主线程微卡顿的风险：

1. **锁竞争风险 (DatabaseManager::m_mutex)**：
    - **现象**：主线程在 `MetadataManager` 中频繁调用 `getMemoryDb` 以获取句柄；同时，后台 `workerLoop` 在执行同步任务时，会调用 `getDiskDb`。
    - **瓶颈**：这两个函数均竞争同一个全局互斥锁 `m_mutex`。
    - **卡顿场景**：若后台磁盘写入由于 OS 层面的文件锁或 I/O 阻塞导致时间稍长，虽然 `getDiskDb` 本身很快，但如果 `DatabaseManager` 的其他同步方法（如 `init` 或 `shutdown`）正在持有该锁，则主线程获取内存库句柄时会发生挂起。
2. **主线程内存库写入负担**：
    - **现象**：`MetadataManager::persistAsync` 在主线程执行 `sqlite3_step` 操作内存库。
    - **瓶颈**：虽然内存库极快，但在执行“大批量标签重命名”等事务时，由于在主线程执行 SQL，仍可能造成毫秒级的 UI 抖动。

## 4. 改进与加固方案

### 4.1 锁分离优化 (针对 DatabaseManager)
- **方案**：将 `m_driveDbs` 的查询逻辑由“实时遍历”改为“读写分离锁（shared_mutex）”或“原子缓存”。
- **目标**：主线程调用 `getMemoryDb` 走 `shared_lock`（共享读），后台线程调用 `getDiskDb` 亦然，彻底杜绝读写竞争导致的 UI 阻塞。

### 4.2 事务原子化下沉
- **方案**：将“内存库写入”与“磁盘库写入”作为一个整体原子任务投递给后台队列。
- **目标**：主线程仅负责修改 `m_cache`（C++ 对象），所有的 SQL 逻辑（无论是内存还是磁盘）均由后台线程完成。这才是真正的“零卡顿”最终形态。
