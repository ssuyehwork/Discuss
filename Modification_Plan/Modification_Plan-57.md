# 架构重构防 AI 脑补与钻空子强制红线声明 —— Modification_Plan-57.md

## 1. 任务背景

本方案承接自 `Modification_Plan-56.md` 的全局架构规划。为了防止未来 AI 执行者在实现 **“方案 A - Qt 模型数据流向上微循环”** 以及 **“命令总线（Command Bus Pattern）”** 的过程中，由于缺乏硬性约束而钻入“默认约定”或“灰色地带”自作聪明（脑补），从而引入新的同步阻塞、信号死循环以及状态丢失等致命 Bug，本方案无条件新建，将防脑补的工程级技术红线和反钻空子守则做终极落地并作为后续重构的强制性约束。

---

## 2. 问题定位

在宏观架构向单一职责（SRP）重构时，AI 在编码期极易出现由于“追求一时跑通”而产生的局部脑补行为。必须对以下 4 个最隐蔽的崩溃与死锁高发漏洞点进行定点阻断：

1. **同步重型 I/O 浸染模型层**：在模型 `setData` 中脑补同步磁盘文件移动，卡死 UI 线程。
2. **槽函数级联陷入信号死循环**：在接收到重命名刷新信号后，过度脑补调用 `loadDirectory()` 全量重新加载，引发 Stack Overflow 瞬间闪退。
3. **命令伪解耦与状态穿透**：Command 类内部直接篡改表现层（UI）的私有状态，导致架构设计名存实亡。
4. **视觉样式参数脑补退化**：在新建交互或命令 Dialog 时脑补组件样式、漏掉原生清除按钮与置顶激活色。

## 4. 详细解决方案

在重写或微调任何代码文件时，未来 AI 执行者必须无条件执行以下“反脑补”底层架构防线：

### 4.1 【反脑补红线一】：禁止模型层（setData）直接阻塞执行重型 I/O
- **硬性约束**：在 `FerrexVirtualDbModel::setData` 内部，**严禁直接出现任何同步的文件重命名、磁盘路径检查、或递归的 QDir 操作**。
- **实施标准**：模型层必须调用非 GUI 异步服务进行操作（即派发给 Command 或是专门的非 GUI 工作线程）。当底层工作线程操作成功后，方可由工作线程将事件同步回主线程模型，通过模型更新其内部容器并对外发射刷新信号（`emit recordRenamed(...)`）。
- **判定红线**：一旦在模型类的代码中发现同步的 `QFile::rename` 或 `QDir` 的 entry 扫描，一律判定为“重型 I/O 侵入主线程”的严重违规脑补，退回重做。

### 4.2 【反脑补红线二】：严禁在重命名信号槽中执行全量 `loadDirectory` 重新扫描
- **硬性约束**：当 `ContentPanel` 监听到模型的 `recordRenamed(oldPath, newPath)` 信号后，**绝对禁止在对应的槽函数中调用 `loadDirectory`、`refreshAll` 或类似的重新全量扫描物理磁盘路径的函数**。
- **实施标准**：为了防止信号连带导致“信号死循环（Stack Overflow）”，重命名成功的刷新必须走**“增量更新/原子级重置”**机制：
  ```cpp
  // 必须只做这三步：
  1. 修改内存ToIndex映射：m_pathToIndex[newPath] = oldIndexRow;
  2. 通知视图该行改变：emit dataChanged(idx, idx, {PathRole, DisplayRole});
  3. UI 重新滚动定位：this->selectAndScrollToPath(newPath);
  ```
  绝对不能去执行全量磁盘对账，禁止触发任何重新读取操作。

### 4.3 【反脑补红线三】：Command 类内部的“零 UI 指针强耦合”约束
- **硬性约束**：在实现右键命令类（如 `SecureDeleteCommand`, `EncryptCommand`）时，**Command 类内部的成员变量严禁包含表现层 `QWidget`、`ContentPanel` 或其直接子控件的任何裸指针或强引用**。
- **实施标准**：
  - Command 只能持有其业务操作所需的物理要素（如 `QStringList paths`, `QString targetDir`）。
  - Command 类的物理结果必须由其通过标准的 Qt 信号抛回给 App 控制器，禁止在 Command 中直接篡改 UI 控件的私有变量（如 `m_isLoading` 或 `refreshVisibleThumbnails()`）。
  - **无 GUI 测试标准**：此命令必须能在完全没有 `QApplication` 的纯控制台 C++ 测试中无伤运行通过。

### 4.4 【反脑补红线四】：新建/重构 UI 组件时强制“无脑补样式对账”
- **硬性约束**：在重构右键关联的 Dialog、重命名输入框或二次确认弹窗时，**AI 严禁自己现场定义任何边框粗细、圆角大小、字号、或者随手拼凑色值**。
- **实施标准**：
  - 新增的可编辑输入框必须强制搭载：`setClearButtonEnabled(true)`。
  - 新建按钮样式悬停背景必须且只能使用：`#3E3E42`（`Style::HoverBackground`）。
  - 新建按钮样式按下背景必须且只能使用：`#4E4E52`（`Style::PressedBackground`）。
  - 关闭按钮默认背景及悬停背景必须且只能持续高亮显示为：`ErrorRed` (`#e81123`)。
  - 边缘留白及容器四周边距基准必须强制靠齐：`kEdgeMargin = 5`（边缘热区感应距离）。

## 6. 实现准则与预警【核心】

1. **编译拦截**：由于方案高度强调了信号槽的解耦（方案 A）和命令解耦，未来执行重构时，头文件中必须完全删除对方的头文件依赖，使用 **前置声明** 来打破头文件循环依赖，防止引发编译器 `invalid use of incomplete type` 的编译中止错误。
2. **防死锁预警**：Command 在操作文件系统和 SCCH 缓存时，必须首先通过 `DatabaseManager` 单锁线程对齐，防止出现两个 Command 交叉占用缓存锁与数据库连接池引起的系统死锁（Deadlock）。
