# 标签权重统计模型与“常用标签”引导功能 —— Analysis_Modification_Plan-82.md

## 1. 任务背景
目前 ArcMeta 的标签系统仅作为静态属性存在，缺乏使用频次的度量。当用户积累了成百上千个标签时，在标签管理视图中快速检索常用标签变得困难。系统缺乏基于“热点数据”的智能引导。

## 2. 问题定位

### 2.1 根因分析
1.  **数据模型缺失**：数据库中虽存储了项目与标签的关联（`item_tags`），但缺乏一个聚合权重的视图或表来快速读取每个标签的使用计数。
2.  **UI 占位未实现**：`TagManagerView.cpp` 中存在 `TODO: 常用标签逻辑`，目前该区域显示为空白，未对接后端数据。

### 2.2 涉及文件
- `src/meta/MetadataManager.cpp`：需提供标签权重统计接口。
- `src/ui/TagManagerView.cpp`：实现“常用标签”区域的动态渲染。

## 4. 详细解决方案

### 4.1 实现后端权重统计
在 `MetadataManager` 或专用的标签仓库中实现权重获取逻辑。

```cpp
// 伪代码：获取前 N 个高频标签
struct TagWeight { std::wstring name; int count; };
std::vector<TagWeight> getTopTags(int limit = 20) {
    // 执行 SQL: 
    // SELECT tag_name, COUNT(*) as usage_count 
    // FROM item_tags 
    // GROUP BY tag_name 
    // ORDER BY usage_count DESC 
    // LIMIT ?;
}
```

### 4.2 填充 TagManagerView
在 `TagManagerView::refresh()` 时，调用上述接口并填充至顶部的流式布局容器中。

1.  **UI 构建**：在现有“标签组”上方插入一个 `FlowLayout` 容器，命名为 `m_popularTagsContainer`。
2.  **视觉区分**：常用标签可以采用略微不同的背景色或带有热度图标（如火苗图标），以区别于普通标签。
3.  **交互**：点击常用标签，逻辑应与点击普通标签一致（即跳转至对应标签的搜索结果视图）。

## 5. 修改边界声明【红线】

**本次方案涉及范围：**
- [ ] `TagManagerView` 的 UI 刷新逻辑。
- [ ] `MetadataManager` 的只读统计接口。

**明确禁止越界修改的范围：**
- [ ] 禁止修改标签的存储结构（不增加冗余的 weight 字段，以 SQL 实时聚合为主以确保数据绝对准确）。
- [ ] 禁止修改标签重命名或合并逻辑。

## 6. 实现准则与预警【核心】

1.  **缓存优化**：由于标签关联表可能达到数十万行，实时 `GROUP BY` 可能会有毫秒级延迟。建议对统计结果进行分钟级缓存或在标签变更时异步触发增量更新。
2.  **空状态处理**：当系统无任何标签或所有标签使用次数为 0 时，应优雅隐藏“常用标签”分区。
3.  **样式对齐**：常用标签的样式必须遵循 `StyleLibrary.h` 中的规范，禁止私自设定突兀的颜色。
