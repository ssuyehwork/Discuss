# 硬盘盘符置顶失效与持久化脱节分析 —— Analysis_Modification_Plan-75.md

## 1. 任务背景
在“此电脑”视图下，用户对物理磁盘盘符执行置顶操作。虽然 UI 瞬时反馈成功，但状态无法持久化，重启或切换视图后置顶失效。

## 2. 深度诊断结论

### 2.1 UI 构建逻辑缺失 (Data Injection Void)
在 `ContentPanel.cpp` 的 `loadDirectory` 函数中，加载 `computer://` 路径的逻辑分支采用硬编码方式生成盘符列表：
* **现状**: `for (const QFileInfo& drive : drives)` 循环中仅填充了 `path` 和 `isDir`。
* **缺陷**: 未调用元数据注入逻辑（如 `createItemRecord` 或 `updateRecordMetadata`），导致 UI 始终显示“裸路径”状态，无法反射数据库中已存的置顶信息。

### 2.2 持久化层循环依赖 (Persistence Paradox)
系统元数据采用“一卷一库”模式：
* **机制**: 路径 `X:\Path` 的元数据存在 `X:\.arcmeta\...` 下。
* **漏洞**: 盘符 `X:\` 本身是“库的容器”。目前的持久化逻辑试图将 `X:\` 的置顶状态存入其自身的卷级库中。
* **失效场景**: 
    1. 若该硬盘未执行过扫描，其 `.arcmeta` 数据库尚未初始化，`getMemoryDb()` 返回空，导致保存指令静默失效。
    2. 这种“自己存自己”的逻辑不符合全局元数据管理常识。

### 2.3 路径归一化歧义 (Normalization Ambiguity)
`MetadataManager::normalizePath` 会将 `C:` 补全为 `C:\` 并转为小写。
* **风险**: 若 `QDir::drives()` 返回的路径字符串与 `setData` 传入的路径在尾斜杠处理上存在微秒级不一致（如 `C:/` vs `c:\`），会导致内存缓存 `m_cache` 的哈希键值碰撞失败。

## 3. 解决方案 (资深程序员视角)

### A. 全局配置重定向 (Config Redirection)
**核心建议**: 盘符置顶状态不应存在卷级库中，而应存在**全局主控库**（AppConfig 或 Main Metadata DB）中。
* **理由**: 硬盘的置顶是全应用级的，不随硬盘内数据的增删而消失，且能解决“库未挂载”时的持久化问题。

### B. UI 构建链路标准化
修改 `loadDirectory` 加载硬盘的逻辑：
```cpp
// 确保盘符加载时同步拉取持久化状态
for (const QFileInfo& drive : drives) {
    driveRecords.push_back(ContentPanel::createItemRecord(QDir::toNativeSeparators(drive.absolutePath())));
}
```

### C. 强化盘符识别与排序
在“此电脑”视图加载末尾强制调用 `m_proxyModel->sort(0)`，并确保 `lessThan` 能够正确处理 `computer://` 下的虚拟排序权重。

## 4. 结论
盘符置顶失效是**UI 加载漏项**与**存储逻辑位级冲突**的综合结果。必须通过“存储层级提升”和“加载链路补全”两步走策略进行修复。
