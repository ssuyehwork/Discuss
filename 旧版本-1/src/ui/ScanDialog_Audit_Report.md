# ScanDialog.cpp 逻辑审计与失败点分析报告

作为资深程序员旁观者，我对 `ScanDialog.cpp` 及其核心引擎 `MftReader.cpp` 进行了深度审计。虽然该模块在 UI 层面能够运行并显示搜索结果，但在逻辑架构和工业级健壮性上存在以下致命失败点：

## 1. 磁盘 I/O 模型的灾难性误用 (OpenFileById)
*   **现象**：在 `MftReader::loadMftDirect` 中，代码尝试通过 `OpenFileById` 获取文件物理大小。
*   **失败逻辑**：
    *   **传参错误**：代码将 `FileReferenceNumber` 的地址强制转换为结构体指针，导致 API 访问非法内存。
    *   **性能瓶颈**：在全盘扫描（百万级文件）时，对每个条目调用此 API 会将原本线性的 MFT 读取变为随机磁盘寻道。
*   **后果**：扫描速度下降数十倍，且由于传参错误，文件大小通常显示为 0 或错误数据。

## 2. UI 线程的同步死锁风险
*   **现象**：`ScanDialog::onTriggerSearch` 同步调用 `MftReader::instance().updateActiveDrives`。
*   **失败逻辑**：该函数内部使用 `QWriteLocker`（写锁）。如果后台搜索线程持有了 `QReadLocker`（读锁）且正在进行重度 I/O 或大循环匹配，UI 线程将被无限期挂起。
*   **后果**：点击“搜索”按钮时，程序频繁出现“无响应”状态。

## 3. 内存管理的“只增不减”漏洞 (String Pool)
*   **现象**：`MftReader` 采用 SoA 架构，文件名存储在全局字符串池中。
*   **失败逻辑**：在 USN 实时更新（重命名或修改）时，如果新名字更长，代码会向池中追加新字符串，但**从未回收**旧字符串的内存。
*   **后果**：长期运行下，内存占用会随系统文件变动而线性增长，最终导致 OOM 崩溃。

## 4. 数据实时性失效 (USN 路径缺陷)
*   **现象**：`MftReader::updateEntryFromUsn` 负责同步文件变动。
*   **失败逻辑**：更新逻辑中唯独漏掉了物理大小（`m_sizes`）的同步。
*   **后果**：用户修改文件内容后，搜索结果列表中的“大小”列依然显示旧数据，数据一致性被破坏。

## 5. 架构组件生命周期的设计悖论
*   **现象**：`ScanDialog` 的析构函数调用了 `MftReader::instance().clear()`。
*   **失败逻辑**：`MftReader` 是单例，应作为全局服务存在。现在的逻辑导致用户只要关闭窗口，所有的内存索引（几百 MB 数据）就会被立即清空。
*   **后果**：用户下次打开搜索框必须重新扫描或重新加载快照，严重违背了“即开即用”的工具设计初衷。

## 6. “幽灵索引”导致的性能衰减
*   **现象**：删除文件仅标记 `m_frns` 为 0。
*   **失败逻辑**：搜索循环（`MftReader::search`）依然会遍历这些已经失效的“死条目”。
*   **后果**：随着系统使用时间的增加，失效条目堆积，搜索性能会单调递减。

## 7. UI 图标缓存的线程安全隐患
*   **现象**：`ScanTableModel` 在 UI 线程直接访问 `ScanDialog` 的 `m_iconCache`。
*   **失败逻辑**：由于 MFT 引擎频繁发送 `dataChanged` 信号，如果异步触发模型刷新，对 `QHash` 的并发读写会导致随机闪退。

---

## 💡 改进建议
1.  **解析 $DATA 属性**：直接从 MFT 记录中读取大小，消除 `OpenFileById` 的额外开销。
2.  **原子掩码**：使用 `std::atomic` 位掩码管理驱动器过滤状态，消除 UI 写锁竞争。
3.  **分块内存管理**：对字符串池引入碎片整理或引用计数机制。
4.  **解耦生命周期**：将索引引擎的生命周期提升至 `Application` 层，实现真正的后台常驻。
